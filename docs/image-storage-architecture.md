# Image Storage Architecture & S3 Migration Plan

## 1. Current State Analysis

### 1.1 Image Generation Pipeline

Images are generated by the **Post Generator** pipeline (Pipeline 2) at `ai-service/app/agents/post_generator/nodes/image_generation.py`.

```
POST /api/v1/posts/generate
  → runner.py → graph.py
      → strategy_alignment
      → content_generation
      → image_prompt_creation  ← GPT-4o generates prompt text
      → image_generation       ← OpenAI gpt-image-1.5 generates PNG bytes
      → auto_review
      → output_packaging       ← writes image bytes to storage, persists DB record
      → END
```

**Image client** (`app/clients/bfl_client.py`):
- Model: `gpt-image-1.5` via `openai.AsyncOpenAI`
- Returns base64-encoded PNG bytes decoded to raw `bytes`
- Content-type: `image/png`
- Supported sizes: `1024x1024`, `1024x1536` (4:5 / 9:16), `1536x1024` (16:9)

**Storage call** in `image_generation.py`:
```python
storage_key = f"posts/{scan_run_id}/images/{post_id}.{ext}"
storage = get_storage()
saved_path = storage.write_bytes(storage_key, result.image_bytes, content_type=result.content_type)
post_copy["image_path"] = saved_path  # stored in state, later written to DB
```

### 1.2 Storage Backend Selection (`app/core/storage.py`)

```
APP_ENV == "production" AND S3_BUCKET set  →  S3Storage
otherwise                                  →  LocalStorage
```

#### Development — LocalStorage

| Property | Value |
|----------|-------|
| Physical location | `ai-service/posts/{scan_run_id}/images/{post_id}.png` |
| `write_bytes` return value | relative key: `posts/{scan_run_id}/images/{post_id}.png` |
| Value stored in `content_posts.image_path` | `posts/{scan_run_id}/images/{post_id}.png` |
| Public URL (for TikTok) | `{STORAGE_PUBLIC_BASE_URL}/posts/{scan_run_id}/images/{post_id}.png` |
| Served by | FastAPI `StaticFiles` mount at `/static` → `ai-service/` |
| ngrok required for TikTok | Yes — TikTok's PULL_FROM_URL must reach a public host |

#### Production — S3Storage (current, partial implementation)

| Property | Value |
|----------|-------|
| S3 key written | `{S3_PREFIX}/posts/{scan_run_id}/images/{post_id}.png` |
| `write_bytes` return value | `s3://{S3_BUCKET}/{S3_PREFIX}/posts/{scan_run_id}/images/{post_id}.png` |
| Value stored in `content_posts.image_path` | `s3://bucket/prefix/posts/…` ← **BUG: full URI, not a raw key** |
| Public URL method | `generate_presigned_url` (1-hour expiry) |
| CloudFront | Not yet configured |

### 1.3 Database Schema

**`ai.content_posts.image_path`** (SQLAlchemy: `String`, nullable):
- Dev: stores relative key → `posts/{scan_run_id}/images/{post_id}.png`
- Prod (current): stores full S3 URI → `s3://bucket/prefix/posts/…` ← inconsistent with the key-based API

**Prisma mirror** (`backend/prisma/schema.prisma`):
```prisma
imagePath  String?  @map("image_path")
```
The NestJS backend reads this field; the raw value is returned to the client in `GET /v1/posts/:id`.

### 1.4 Publish Flow — How the Image URL is Resolved

In `app/agents/publish_post/graph.py → resolve_and_validate_node`:
```python
storage = get_storage()
image_key = post.image_path or ""          # reads DB value
if not image_key or not storage.exists(image_key):
    raise ValueError("Thumbnail not found")
image_public_url = storage.get_public_url(image_key)
```

The `image_public_url` is then passed to `tiktok.init_photo_post(photo_urls=[image_url])`.
TikTok uses **PULL_FROM_URL** — it downloads the image from that URL at publish time.

### 1.5 API Exposure (Gap)

`PostDetail` schema (`app/api/v1/schemas/post.py`) is **missing `image_path`**. The field exists in the DB and Prisma model but is not included in the FastAPI response schema, so the frontend cannot display post images through the ai-service API.

The NestJS `ContentPost` model (`posts.controller.ts → detail()`) returns the full Prisma row which includes `imagePath`, so the field is available to the frontend through the NestJS gateway — but its value is a storage key or S3 URI, not a usable URL.

## 2. Known Bugs

### Bug 1 — S3Storage double-prefix on read-back

`S3Storage.write_bytes()` returns `s3://{bucket}/{prefix}/{key}` and this full URI is stored as `image_path` in the DB. Later, `resolve_and_validate_node` passes this URI back to `storage.exists(image_key)` and `storage.get_public_url(image_key)`. Both methods call `_full_key(key)` which prepends the prefix **again**:

```python
def _full_key(self, key: str) -> str:
    return f"{self.prefix}/{key}"   # → "prefix/s3://bucket/prefix/..."
```

**Result**: `exists()` always returns `False` in production, causing every publish attempt to fail with "Thumbnail not found".

**Fix**: `write_bytes` (and `write_text`) must return only the raw `key`, not the full `s3://` URI. The `s3://` URI is an internal detail.

### Bug 2 — 1-hour presigned URL expiry for TikTok

TikTok downloads the image when `init_photo_post` is called. If the presigned URL expires before TikTok actually fetches the file (e.g., in a scheduled/golden-hour post), the publish will fail with a permission error. CloudFront public URLs do not expire and are the correct solution.

### Bug 3 — `image_path` not in `PostDetail` API schema

The frontend cannot display images via the ai-service API endpoint. The NestJS gateway returns `imagePath` but it is an opaque storage key, not a usable URL.

## 3. S3 Integration Plan

### 3.1 S3 Bucket Structure

```
s3://marketing-content-media-{env}/
├── trending-scanner/                   ← S3_PREFIX
│   ├── posts/
│   │   └── {scan_run_id}/
│   │       ├── images/
│   │       │   └── {post_id}.png       ← generated post images
│   │       ├── {post_id}.json          ← individual post JSON
│   │       └── output.json             ← full pipeline output
│   ├── reports/
│   │   └── {scan_run_id}/
│   │       └── report.md
│   └── strategy/
│       └── {scan_run_id}/
│           └── strategy_update.json
```

### 3.2 Fix `S3Storage.write_bytes` Return Value

**File**: `ai-service/app/core/storage.py`

```python
# BEFORE (bug): returns full s3:// URI
def write_bytes(self, key: str, data: bytes, content_type: str) -> str:
    full_key = self._full_key(key)
    self.client.put_object(...)
    return f"s3://{self.bucket}/{full_key}"  # ← wrong

# AFTER (fix): returns raw key (same contract as LocalStorage)
def write_bytes(self, key: str, data: bytes, content_type: str) -> str:
    full_key = self._full_key(key)
    self.client.put_object(...)
    return key  # ← consistent with LocalStorage

# Same fix for write_text
```

**DB migration**: If the production DB already has `s3://` URIs in `image_path`, a one-off migration script must strip the `s3://{bucket}/{prefix}/` prefix to recover the raw key.

### 3.3 CloudFront Distribution

Place a CloudFront distribution in front of the S3 bucket to:
- Provide stable (non-expiring) public URLs for TikTok PULL_FROM_URL
- Cache images at edge for low-latency frontend display
- Avoid exposing S3 bucket directly

```
CloudFront domain:    media.yourdomain.com (or XXXX.cloudfront.net)
Origin:               marketing-content-media-prod.s3.ap-southeast-1.amazonaws.com
Origin path:          /trending-scanner
Cache behavior:       Cache images for 30 days (Cache-Control: max-age=2592000)
```

**S3 bucket policy** (OAC — Origin Access Control):
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowCloudFrontServicePrincipal",
    "Effect": "Allow",
    "Principal": {
      "Service": "cloudfront.amazonaws.com"
    },
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::marketing-content-media-prod/*",
    "Condition": {
      "StringEquals": {
        "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT_ID:distribution/DISTRIBUTION_ID"
      }
    }
  }]
}
```

No public S3 bucket ACLs needed — CloudFront OAC handles access.

### 3.4 Update `get_public_url` for S3

```python
class S3Storage(StorageBackend):
    def __init__(self, bucket: str, region: str, prefix: str, cloudfront_url: str = "") -> None:
        self.cloudfront_url = cloudfront_url.rstrip("/")
        ...

    def get_public_url(self, key: str) -> str:
        if self.cloudfront_url:
            # Stable CloudFront URL — does not expire
            return f"{self.cloudfront_url}/{key}"
        # Fallback: presigned URL (development / no CloudFront)
        full_key = self._full_key(key)
        return self.client.generate_presigned_url(
            "get_object",
            Params={"Bucket": self.bucket, "Key": full_key},
            ExpiresIn=3600,
        )
```

New config var in `app/config.py`:
```python
CLOUDFRONT_URL: str = ""   # e.g. https://media.yourdomain.com
```

Updated `get_storage()`:
```python
return S3Storage(
    bucket=settings.S3_BUCKET,
    region=settings.S3_REGION,
    prefix=settings.S3_PREFIX,
    cloudfront_url=settings.CLOUDFRONT_URL,
)
```

### 3.5 Add `image_url` to API Responses

**`app/api/v1/schemas/post.py`** — add `image_url` to `PostDetail`:
```python
class PostDetail(BaseModel):
    ...
    image_url: str | None = None        # computed public URL
    image_path: str | None = None       # raw storage key (for internal use)
```

**`app/api/v1/posts.py`** — compute URL on read:
```python
from app.core.storage import get_storage

@router.get("/{post_id}", response_model=PostDetail)
async def get_post(post_id: uuid.UUID, ...):
    post = ...  # query DB
    detail = PostDetail.model_validate(post)
    if post.image_path:
        storage = get_storage()
        detail.image_url = storage.get_public_url(post.image_path)
    return detail
```

### 3.6 Backend (NestJS) — Image URL Resolution

The NestJS gateway reads `imagePath` from Prisma. Instead of exposing the raw key, compute the public URL server-side:

**Option A (recommended)**: Add an `imageUrl` computed field in `posts.controller.ts`:
```typescript
import { ConfigService } from '@nestjs/config';

// In detail() and list():
const cloudfrontUrl = this.config.get<string>('CLOUDFRONT_URL', '');
const rows = items.map(row => ({
  ...row,
  imageUrl: row.imagePath && cloudfrontUrl
    ? `${cloudfrontUrl}/${row.imagePath}`
    : null,
}));
```

**Option B**: Frontend calls `GET /v1/posts/:id` which already returns `imagePath`, then constructs the URL using the known CloudFront domain from a public env var (`NEXT_PUBLIC_MEDIA_URL`).

Option B is simpler for a thesis project; Option A is better for production (no sensitive URL patterns in client code).

### 3.7 Frontend — Image Display

In `frontend/src/lib/api/types.ts`:
```typescript
export interface ContentPost {
  ...
  imagePath?: string | null;
  imageUrl?: string | null;   // preferred: use this for display
}
```

In components, use `imageUrl` if available, else fall back to constructing from `imagePath`:
```tsx
const src = post.imageUrl
  ?? (post.imagePath ? `${process.env.NEXT_PUBLIC_MEDIA_URL}/${post.imagePath}` : null);
```

### 3.8 S3 CORS Configuration

Required for browser-side image display (if using presigned URLs):
```json
[{
  "AllowedHeaders": ["*"],
  "AllowedMethods": ["GET", "HEAD"],
  "AllowedOrigins": ["https://yourdomain.com", "http://localhost:3001"],
  "ExposeHeaders": ["ETag"],
  "MaxAgeSeconds": 3000
}]
```

With CloudFront, CORS is only needed if the frontend fetches images with credentials. Standard `<img>` tags don't require CORS headers.

### 3.9 IAM Role for ECS Tasks

The ai-service ECS task needs this policy on the S3 bucket:
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "s3:PutObject",
      "s3:GetObject",
      "s3:DeleteObject",
      "s3:HeadObject"
    ],
    "Resource": "arn:aws:s3:::marketing-content-media-prod/trending-scanner/*"
  }]
}
```

Use **IAM Roles for ECS Tasks** (not access keys). The `boto3` S3 client automatically picks up the task role via the metadata endpoint.

## 4. Complete Image Data Flow (Post-Fix)

### 4.1 Generation Flow

```
POST /v1/posts/generate  (frontend → NestJS → ai-service)
  ↓
image_generation_node
  ↓  OpenAI gpt-image-1.5
  ← PNG bytes (base64-decoded)
  ↓
S3Storage.write_bytes(
    key="posts/{scan_run_id}/images/{post_id}.png",
    data=png_bytes,
    content_type="image/png"
)
  ↓  boto3 put_object → S3 key: "trending-scanner/posts/…/images/{id}.png"
  ← returns raw key: "posts/{scan_run_id}/images/{post_id}.png"
  ↓
DB: content_posts.image_path = "posts/{scan_run_id}/images/{post_id}.png"
```

### 4.2 Display Flow

```
GET /v1/posts/{id}  (frontend → NestJS)
  ↓
NestJS → Prisma: SELECT image_path FROM ai.content_posts
  ← "posts/{scan_run_id}/images/{post_id}.png"
  ↓
NestJS computes:
  imageUrl = "{CLOUDFRONT_URL}/posts/{scan_run_id}/images/{post_id}.png"
  ← returns imageUrl in response JSON
  ↓
Frontend: <img src={post.imageUrl} />
  ↓  browser HTTP GET
CloudFront edge ← cache hit → PNG bytes
              ← cache miss → S3 fetch → cache → PNG bytes
```

### 4.3 Publish Flow

```
POST /v1/publish/{post_id}  (frontend → NestJS → ai-service)
  ↓
resolve_and_validate_node
  ← post.image_path = "posts/{scan_run_id}/images/{post_id}.png"
  ↓
S3Storage.exists("posts/{scan_run_id}/images/{post_id}.png")
  → HeadObject: "trending-scanner/posts/…/images/{id}.png" → 200 OK
S3Storage.get_public_url("posts/{scan_run_id}/images/{post_id}.png")
  → "https://media.yourdomain.com/posts/{scan_run_id}/images/{post_id}.png"
  ↓
tiktok.init_photo_post(photo_urls=["https://media.yourdomain.com/…"])
  ↓  TikTok PULL_FROM_URL downloads the image
  ← publish_id
  ↓
poll_publish_status → PUBLISH_COMPLETE
```
